    World Generation Pipeline — Galaxy → Systems → Objects¶

    Overview¶

    The procedural world generation system creates an entire galaxy from a single seed, descending through four levels of abstraction: Galaxy (global bounds and density) → Systems (individual star systems with deterministic content) → Objects (3D meshes placed in the scene). Every level is fully deterministic — the same seed always produces identical results.

    The pipeline follows this flow: 

    GalaxyData (seed, bounds, density curve)
    ↓ generates grid coordinates
GalaxySystemData[] (one per grid cell that passes density check)
    ↓ SystemGenerator.generate_system_content()
SystemRecipe (planets, stations, enemies, POIs, megastructures)
    ↓ ProceduralLevelBuilder.build_level()
3D scene graph (MeshInstance3D, GPUParticles3D, OmniLight3D, etc.)

    GalaxyData — Global Configuration¶

    Located at /home/j/3d-space/scripts/resources/galaxy_data.gd (244 lines), GalaxyData is a Resource that defines the entire galaxy's structure. It serves as the root configuration for all procedural generation.

    Core Parameters¶

    Parameter Type Default Description
    galaxy_seed int 12345 Master seed — same value = identical galaxy
    galaxy_name String "Milky Way" Display name
    galaxy_bounds Vector3i (10,10,10) Grid dimensions → 10×10×10 = 1000 possible system locations
    system_spacing float 100.0 Distance between grid points in light-years
    system_position_jitter float 0.35 Per-axis offset as fraction of spacing (±35%)
    galaxy_warp_strength float 0.65 Large-scale warp amount (0 = rigid cube, 1 = full twist/noise/disk)
    connectivity_radius float 150.0 Max jump drive range in light-years
    min_systems int 100 Minimum systems to generate

    Galaxy Warp System (_apply_large_scale_warp)¶

    Transforms a rigid grid into an organic galactic shape through four sequential operations:

    base_pos = Vector3(coords) * system_spacing
    ↓
1. Disk flattening — Y-axis compressed toward center using galaxy_disk_vertical_scale (0.55 default)
2. Radial twist — XZ plane rotated by normalized_radius × galaxy_warp_twist_degrees (65° default)
3. Arm banding — sin(angle × arm_count + radius × 8) pushes positions radially (arm_amplitude = 0.45, arm_count = 4)
4. Noise displacement — Three independent FastNoiseLite instances (SIMPLEX type) add organic variation

    The warp is applied per-coordinate and cached in _world_position_cache for performance. Jitter is added after warping using a deterministic RNG seeded from get_system_seed(coords).

    Density Curve¶

    A Curve resource where X = normalized distance from center (0–1) and Y = probability multiplier. Default curve: [0, 1.2] → [0.5, 1.0] → [1.0, 0.8], creating slightly denser core and sparser edges. During generation, each grid cell's density is sampled at its normalized distance; if the random value falls below the curve sample, a system is created there.

    Deterministic Seeding¶

    func get_system_seed(coords: Vector3i) -> int:
    var hash_input = "%d,%d,%d,%d" % [coords.x, coords.y, coords.z, galaxy_seed]
    return hash(hash_input)
    This ensures every system gets a unique but reproducible seed derived from its grid position and the master galaxy seed.

    Coordinate Conversion¶

    Method Purpose
    coordinate_to_world(coords) Grid → world position (with warp + jitter, cached)
    coordinate_to_raw_world(coords) Grid → raw world position (spacing only, no warp — for density sampling)
    world_to_coordinate(world_pos) World → nearest grid point
    is_coordinate_valid(coords) Bounds check

    GalaxySystemData — Per-System Metadata¶

    Located at /home/j/3d-space/scripts/resources/galaxy_system_data.gd (384 lines), this lightweight resource (~100 bytes per system) stores all data needed to deterministically generate a system's content. The actual 3D objects are generated on-demand from the seed, not stored directly.

    Star Types¶

    enum StarType {
    O, B, A, F, G, K, M,           # Main sequence (blue giant → red dwarf)
    WHITE_DWARF, NEUTRON, BLACK_HOLE,  # Stellar remnants
    DARK_STAR                         # Hypothetical dark matter star
}

    Economy Types¶

    enum EconomyType {
    NONE, AGRICULTURAL, EXTRACTION, REFINERY,
    INDUSTRIAL, HIGH_TECH, TOURISM, SERVICE, MILITARY, PRISON
}

    Population Scale (logarithmic)¶

    Value Description
    0 Uninhabited
    1–3 Outpost → Settlement → Colony
    4–6 Town → City → Metropolis
    7–10 Millions → Tens of M → Hundreds of M → Billions

    Name Generation¶

    Three deterministic naming schemes based on the system seed: 1. Greek prefix + constellation — "Alpha Centauri", "Beta Cygni" (24 prefixes × 20 constellations) 2. Catalog number — "HIP 12345", "HD 67890" (5 catalog prefixes, 1000–99999 range) 3. Constellation + suffix — "Eridani Prime", "Lyrae Secundus" (20 constellations × 20 suffixes)

    Serialization¶

    Full to_dictionary() / from_dictionary() support for save file persistence, including discovered/scanned/visited flags and handcrafted system overrides.

    SystemGenerator — Content Generation Engine¶

    Located at /home/j/3d-space/scripts/level/system_generator.gd (3192 lines), this RefCounted class is the core generation engine. It takes a GalaxySystemData + GalaxyData and produces a complete SystemRecipe.

    Generation Pipeline¶

    func generate_system_content(system_data: GalaxySystemData, galaxy_data: GalaxyData) -> SystemRecipe:
    1. Create RNG from system_data.system_seed
    2. Get faction data for this system (via ProceduralFactionAssigner)
    3. Calculate effective security (border zone effects)
    4. Create recipe from system data
    5. Generate star properties (+ optional binary companion, 0.1% chance)
    6. Generate planets (2–8 based on star type and habitable zone)
    7. Generate asteroid belts (0–2, avoiding planet orbits)
    8. Generate nebulae
    9. Generate stations (if population > 0, faction-prefixed)
    10. Generate enemy spawns (if effective_security < 0.5)
    11. Generate POIs
    12. Generate megastructures

    Planet Generation (_generate_planets)¶

    Creates 2–8 planets with increasing orbital distances: - Inner planets (i=0): 0.45–0.95 AU - Second planet (i=1): 1.15–2.35 AU - Outer planets (i≥2): 2.2 + i × (2.8–8.8) AU, capped at 30 AU

    Planet type is determined by orbit position relative to the star's habitable zone: | Type | Orbit Range | Radius (Earth radii) | Atmosphere | |------|-------------|---------------------|------------| | LAVA | Very close | 0.8–1.3 | Toxic (30% chance) | | ROCKY | Habitable zone | 0.5–1.5 | Standard/Thin (60% chance) | | OCEAN | Inner habitable | 0.9–1.8 | Standard (always) | | DESERT | Near habitable | 0.7–1.2 | Thin (50% chance) | | TOXIC | Outer hot | 0.9–1.4 | Toxic (always) | | GAS_GIANT | Far out | 3.0–11.0 | Thick (always), 40% ring chance, 2–8 moons | | ICE | Very far | 0.8–2.0 | Thin (40% chance), 1–2 moons |

    Each planet gets: surface gravity (mass/radius²), surface temperature (orbit + star temp + atmosphere), orbital period (Kepler's third law: √(AU³) × 365.25 days), rotation period, axial tilt, resource deposits, and a 10% chance of being inhabited if in the habitable zone.

    Asteroid Belt Generation (_generate_asteroid_belts)¶

    Creates 0–2 belts with collision avoidance: - Radial width: 0.35–1.2 AU - Thickness: 0.08–0.25 AU - Orbit: 1.8–38 AU, checked against all planets (min gap = max(0.7, belt_width × 0.5 + 0.3)) and other belts - Density: 0.4–0.95

    Binary Star System (_generate_binary_star_data)¶

    0.1% chance per system. Companion star type is weighted toward M-dwarfs (94%) with rare O/B/A types. Companion mass/radius/luminosity are scaled relative to the primary. Minimum planet orbit is increased to binary_separation × 1.8 + 0.2 AU for stability.

    Station Generation (_generate_faction_stations)¶

    Only generated if system population > 0. Station count and type are influenced by faction preferences, economy type, and effective security level.

    Enemy Spawn Generation (_generate_faction_enemies)¶

    Generated when effective_security < 0.5. Enemy presence scales inversely with security — lower security means more enemies. Faction affiliation determines enemy ship types and loadouts.

    StarGenerator — 3D Star Visuals¶

    Located at /home/j/3d-space/scripts/level/star_generator.gd (1085 lines), this Node creates the visual representation of stars including: - Regular stars (O through M): Scaled sphere with emissive material, directional + omni lights - Exotic stars: Black hole (accretion disk + event horizon), neutron star (pulsing beacon shader), dark star (particle cloud)

    Star LOD System¶

    Distance Effect
    < lod_close_distance_min (11,000 units) Full detail shell, billboard glow at full intensity
    < lod_medium_distance_min (42,000 units) Medium detail, reduced glow
    > 42,000 units Simplified sphere only

    Visual Components¶

    • Core: Emissive sphere with star_core_emissive.gdshader
    • Detail shell: Outer layer with star_detail_shell.gdshader (animated noise)
    • Billboard glow: 2D sprite facing camera with star_billboard_glow.gdshader, fades between 7,000–18,000 units
    • Proximity lights: OmniLight3D + DirectionalLight3D scaled by star luminosity

    ProceduralLevelBuilder — 3D Scene Construction¶

    Located at /home/j/3d-space/scripts/level/procedural_level_builder.gd (2090 lines), this Node takes a SystemRecipe and instantiates all 3D objects into the scene graph. It's organized as sequential build phases:

    Build Order¶

    1. Planets (PlanetGeneratorService) — with LOD levels, landable planet support
2. Gates (gate.tscn) — placed at GATE_RADIUS = 100,000 units from center
3. Asteroid belts (asteroid_field.gd)
4. Stations (imported GLB models with albedo tint + metallic/roughness overrides)
5. Enemy spawns (npc_ship.tscn with variant resolver)
6. POIs (ProceduralPOIBuilder — see below)
7. Megastructures (12 types, see SystemRecipe.MegastructureKind enum)

    Scale Configuration¶

    Parameter Default Description
    orbit_au_to_units 28,000 1 AU = 28,000 game units (reduced from 149.6M km for gameplay)
    planet_radius_units_per_earth 2,400 Earth radius in game units (raised from 50 to 150 — 3× scale)
    max_orbit_au 50 Maximum orbital distance for planets
    hero_planet_max_lod_level 16 Full detail LOD for hero planets (~800ms load)
    background_planet_max_lod_level 12 Reduced LOD for background planets (~100ms load)

    Megastructure Types (12 total)¶

    enum MegastructureKind {
    HIGHWAY, PLANET_RING, HABITAT_RING, SHIPYARD, SOLAR_ARRAY,
    DYSON_SWARM, DYSON_SHELL_SEGMENT, AQUEDUCT, SENSOR_ARRAY,
    DEFENSE_PLATFORM, ORBITAL_TETHER, SATELLITE_CONSTELLATION
}

    Each megastructure type has its own builder method (e.g., _build_planet_ring_megastructure) and dedicated script file. They're placed in a Megastructures container node under the system root.

    ProceduralPOIBuilder — POI Visual Generation¶

    Located at /home/j/3d-space/scripts/level/procedural_poi_builder.gd (1783 lines), this RefCounted class uses a registry pattern to generate 3D visuals for each POI type. Each POI type registers a builder function that returns a Node3D with its mesh hierarchy.

    POI Types¶

    Type Color Description
    ANOMALY Purple (0.7, 0.4, 0.9) Energy anomalies
    DERELICT Brown-gold (0.7, 0.6, 0.45) Abandoned ships/structures
    SATELLITE Blue (0.45, 0.7, 0.9) Orbital satellites
    RESOURCE_CACHE Green (0.4, 0.85, 0.5) Mined resource deposits
    SCIENTIFIC Cyan-blue (0.35, 0.75, 0.9) Research stations
    NAVIGATIONAL Gold (0.9, 0.8, 0.4) Navigation beacons with animated rings
    WARNING Red (0.9, 0.35, 0.35) Hazard warnings

    Shared Resources¶

    • Meshes: Pre-allocated PrimitiveMesh instances shared across all POI instances of the same type
    • Materials: One StandardMaterial3D per POI type, cached at init time
    • Builder registry: Dictionary mapping POIType enum → callable(seed) → Node3D

    Animated Navigational Beacons¶

    NAVIGATIONAL POIs feature a three-ring animation system with sequential pulse effect: each ring pulses in sequence (1.2s offset), creating a "chasing light" effect around the beacon crown. The animation is implemented via an inline GDScript attached at build time.

    ProceduralFactionAssigner — Territory Assignment¶

    Located at /home/j/3d-space/scripts/level/procedural_faction_assigner.gd (35 lines), this RefCounted class provides O(1) faction lookup using a pre-computed coordinate hash map. Territories are computed once from the galaxy seed and stored as {Vector3i → faction_id} mappings.

    func configure_from_territories(territories: Array) -> void:
    for territory in territories:
        for coords in territory.system_coords:
            _coords_to_faction[coords] = territory.faction_id

    GalaxyHierarchy — Spatial Partitioning¶

    Located at /home/j/3d-space/scripts/level/galaxy_hierarchy.gd (108 lines), this RefCounted class provides utility helpers for converting system coordinates into hierarchical IDs:

    system_coords → sector_coords → quadrant_coords → region_coords
     ↓                ↓               ↓              ↓
  "sector:x,y,z"  "quadrant:x,y,z"  "region:x,y,z"

    Each level uses configurable sizes from GalaxyHierarchyConfig: - sector_size — systems per sector (X, Y, Z) - sectors_per_quadrant — sectors per quadrant - quadrants_per_region — quadrants per region

    ID parsing is supported via try_parse_*_id() methods for reverse lookups.

    Data Flow Summary¶

    GalaxyData (seed=12345, bounds=(10,10,10))
    │
    ├─→ ProceduralFactionAssigner.configure_from_territories()  [pre-computed territories]
    │
    └─→ For each valid grid cell:
            │
            ├─→ GalaxyData.get_system_seed(coords) → system_seed
            │
            ├─→ GalaxySystemData.new() with coords, seed, star_type, economy, security
            │
            └─→ SystemGenerator.generate_system_content(system_data, galaxy_data):
                    │
                    ├─→ StarData.from_star_type(star_type) + optional binary companion
                    ├─→ _generate_planets() → 2–8 planets with types, orbits, moons
                    ├─→ _generate_asteroid_belts() → 0–2 belts
                    ├─→ _generate_nebulae() → nebula data
                    ├─→ _generate_faction_stations() → stations (if populated)
                    ├─→ _generate_faction_enemies() → enemies (if low security)
                    ├─→ _generate_pois() → points of interest
                    └─→ _generate_megastructures() → megastructure data
                            │
                            ↓ SystemRecipe
                    ProceduralLevelBuilder.build_level(recipe, root_node):
                        1. build_planets() — PlanetGeneratorService with LOD levels
                        2. build_gates() — gate.tscn at GATE_RADIUS
                        3. build_asteroid_belts() — asteroid_field.gd instances
                        4. build_stations() — imported GLB models
                        5. build_enemies() — npc_ship.tscn with variant resolver
                        6. build_pois() — ProceduralPOIBuilder registry
                        7. build_megastructures() — 12 megastructure types
