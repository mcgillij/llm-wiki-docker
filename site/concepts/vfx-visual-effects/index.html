    VFX & Visual Effects System — Thruster Sockets, ShipVFX Manager & Flight Sync¶

    Overview¶

    The VFX system provides real-time thruster trails, booster effects, and glow lighting for all ships in 3dSpace. It follows a manager → socket architecture where ShipVFX acts as the central coordinator that computes emission values from throttle/boost state and broadcasts them to individual ThrusterSocket instances. Each socket independently controls its own particle systems (GPUParticles3D) and glow nodes, enabling per-engine customization while maintaining unified visual behavior.

    ShipVFX — Central VFX Manager¶

    Located at /home/j/3d-space/scripts/vfx/ship_vfx.gd (600 lines), ShipVFX is a @tool Node3D that serves as the master controller for all ship visual effects. It reads throttle, boost, and travel mode state from the flight controller and computes per-socket emission values using configurable presets.

    Architecture¶

    FlightController → set_throttle() / set_boost() / set_travel_mode()
    ↓
ShipVFX._apply_state() — computes emission, glow_color, glow_energy, particle_scale
    ↓
For each ThrusterSocket: call("apply_state", mapped_throttle, boost, emission, glow_color, glow_energy, particle_scale)

    Key Properties¶

    Property Type Default Description
    preset ShipVFXPreset null Visual preset defining colors, multipliers, LOD distances
    sockets Array[NodePath] [] Explicit socket paths (fallback: auto-discovery)
    auto_duplicate_materials bool true Duplicate materials per-instance to prevent shared-state bugs
    use_environment_modifiers bool true Reduce emission in foggy environments
    always_visible_for_subviewports bool true Keep VFX visible in editor subviewport previews
    subviewport_lod_bias float 1.0 Multiplier for LOD distance calculations in subviewports
    preview_emission_boost float 1.4 Emission multiplier when _preview_mode is active (editor only)
    always_full_for_local_player bool true Always render full VFX for the local player's ship
    use_distance_culling bool true Reduce/disable VFX based on camera distance

    State Variables¶

    var _throttle: float = 0.0          # Clamped 0-1, drives emission intensity
var _boost: bool = false            # Boost active → emission × boost_mult
var _travel_mode: bool = false      # Travel mode → emission × boost_mult × 1.5 + blue tint
var _preview_mode: bool = false     # Editor-only preview (safety check in _ready)
var _environment_density: float = 0.0  # Fog density from environment modifier
var _socket_nodes: Array[Node] = []   # Resolved ThrusterSocket instances

    Socket Discovery¶

    _gather_sockets() uses a three-tier discovery strategy: 1. Explicit paths — Nodes listed in @export var sockets: Array[NodePath] 2. Self children — find_children("", "ThrusterSocket", true, false) and "LineThrusterSocket" on self 3. Parent children — Same search on parent node (handles cases where ShipVFX is a child of the ship root)

    Each discovered socket must have an apply_state() method to be accepted into _socket_nodes. Invalid/null instances are pruned by _prune_invalid_sockets().

    Emission Computation Pipeline (_apply_state)¶

    The emission calculation follows this exact sequence:

    1. mapped_throttle = preset.throttle_curve.sample(_throttle)  # Custom curve or identity
2. emission_base = preset.emission_base (default 1.0 if no preset)
3. emission = mapped_throttle × emission_base
4. If _boost: emission *= preset.emission_boost_multiplier (default 5.0)
5. If _travel_mode: emission *= boost_mult × 1.5  # Even brighter than boost
6. lod_multiplier = _get_lod_multiplier() → emission *= lod_multiplier
7. glow_color = preset.glow_color (white default)
   - If _boost: use preset.boost_glow_color (orange default)
   - If _travel_mode: Color(0.2, 0.5, 1.0, 1.0) — blue tint
8. glow_energy = preset.glow_energy × emission
   - If _boost: *= preset.boost_glow_energy_multiplier (default 2.5)
   - If _travel_mode: *= 2.0
9. particle_scale = preset.particle_scale (default 1.0)
   - If _boost: *= preset.boost_particle_scale_multiplier (default 1.4)
   - If _travel_mode: *= 2.0
10. If _preview_mode: emission ×= preview_emission_boost, glow_energy ×= preview_emission_boost
11. If environment_density > 0: emission ×= clamp(1.0 - density, 0.2, 1.0)
12. Enforce min_emission = max(preset.min_emission, node-level min_emission)
13. Debug fallback: if preset == null and use_debug_fallback → emission = debug_fallback_emission (0.35), glow_energy = debug_fallback_glow_energy (14.0)

    LOD System (_get_lod_multiplier)¶

    Distance-based VFX reduction to optimize rendering for distant ships:

    Distance Range Multiplier Effect
    0 – lod_mid_distance (200m default) 1.0 Full VFX
    lod_mid – lod_far_distance (600m default) 0.5 Half emission/particles
    lod_far – lod_cull_distance (800m default) 0.15 Minimal VFX
    > lod_cull_distance 0.0 Culled entirely

    LOD distances are configurable per-preset (lod_mid_distance, lod_far_distance, lod_cull_distance). Manual override via force_lod(level) accepts: "near" (1.0), "mid" (0.5), "far" (0.15), "cull" (0.0).

    Environment Modifiers¶

    set_environment_modifier(density, color, exposure_bias) allows the environment system to reduce VFX intensity in foggy/dense atmospheres: - density (0–1): Fog density; emission × clamp(1 - density, 0.2, 1.0) - Applied only when use_environment_modifiers is true

    Debug System¶

    ShipVFX includes extensive debug capabilities: - trace_debug flag enables verbose logging at every socket application step - set_debug_override(force_visible, show_marker) forces all sockets to maximum visibility and spawns temporary emissive markers (3D sphere + 2D screen overlay) at each socket position - _spawn_temporary_marker(pos, duration=3.0) creates a bright orange MeshInstance3D + CanvasLayer ColorRect at the given world position for visual debugging

    ThrusterSocket — Per-Engine Particle Controller¶

    Located at /home/j/3d-space/scripts/vfx/thruster_socket.gd (860 lines), ThrusterSocket is a @tool Node3D that manages individual thruster particle systems and glow nodes. Each socket represents one engine nozzle on a ship and contains: - Thruster particles — Continuous GPUParticles3D for normal thrust trails - Booster particles — GPUParticles3D for boost-mode effects (more aggressive) - Glow node — Either an OmniLight3D or MeshInstance3D with emissive material

    Node Resolution (_resolve_nodes)¶

    On _ready(), each socket resolves its child nodes by path or name: 

    _thruster_particles = get_node_or_null(thruster_particles_path) as GPUParticles3D
    // fallback: find_child("ThrusterParticles")
_booster_particles = get_node_or_null(booster_particles_path) as GPUParticles3D
    // fallback: find_child("BoosterParticles")
_glow_node = get_node_or_null(glow_node_path)
    // fallback: find_child("ThrusterGlow")

    Auto-Registration with ShipVFX¶

    At runtime (not in editor), each ThrusterSocket automatically walks up the parent chain to find its nearest ShipVFX ancestor and registers itself via node.add_socket(self). This eliminates manual socket path configuration for most ship setups.

    Particle Configuration¶

    Each particle system stores base parameters on resolution: 

    _base_thruster_amount = _thruster_particles.amount
_base_thruster_speed = _thruster_particles.speed_scale
_base_thruster_preprocess = _thruster_particles.preprocess
_base_thruster_scale = _thruster_particles.scale

    World-space enforcement: If particles have local_coords = true, the socket automatically switches them to world-space (local_coords = false), resets preprocess/explosiveness, and restarts. This ensures exhaust trails remain visible behind moving ships rather than being clipped by local coordinate bounds.

    Material Caching (_init_cached_resources)¶

    For performance optimization, each socket caches: - Whether process materials support color_ramp (GradientTexture2D) - Cached Gradient + GradientTexture2D instances for fast color updates - Draw pass 1 material references for emission tinting

    This avoids repeated property list scans on every frame update.

    apply_state() — Per-Socket Visual Update¶

    Called by ShipVFX every frame with computed values:

    func apply_state(
    mapped_throttle: float,      # Curve-mapped throttle (0-1)
    boost: bool,                 # Boost active flag
    emission: float,             # Final emission intensity
    glow_color: Color,           # Tint color for particles and glow
    glow_energy: float,          # Emission energy multiplier
    particle_scale: float        # Particle size scale factor
) -> void

    Thrust particles: - amount = max(100, int(base_amount × emission)) — More throttle = more particles - speed_scale = max(0.1, base_speed × (0.35 + emission)) — Faster exhaust at higher throttle - scale = base_scale × particle_scale — Size scaling from VFX manager - On first emit: sets preprocess=0.0, explosiveness=0.0, snaps global_transform to socket position, restarts

    Booster particles: (only when _boost == true) - amount = max(100, int(base_amount × emission × 2.8)) — 2.8× more particles during boost - speed_scale = max(0.1, base_speed × (1.0 + emission × 1.8)) — Faster exhaust - initial_velocity_min/max on process material for explosive look - scale = Vector2(particle_scale, particle_scale) on process material

    Glow node: - If OmniLight3D: sets light_color = glow_color, light_energy = energy × local_glow_multiplier - If MeshInstance3D with StandardMaterial3D: sets emission = glow_color, emission_energy_multiplier = max(0, energy)

    Color ramp optimization: When process materials support color_ramp, the socket uses cached GradientTexture2D instances for fast per-frame color updates instead of creating new gradients.

    Debug Features¶

    • debug_force_visible: Forces particles to maximum amount (300), scale (4×), and glow multiplier (8×)
    • debug_show_marker: Spawns an emissive sphere at the socket position for visual debugging
    • _spawn_runtime_test_particles(): Creates a temporary GPUParticles3D at the socket position to verify rendering pipeline

    Booster-Only Sockets¶

    The is_booster_only flag allows sockets that only emit during boost mode (e.g., afterburner nozzles). When false and not boosting, emission is set to 0.

    ShipVFXPreset — Visual Configuration Resource¶

    Located at /home/j/3d-space/scripts/resources/ship_vfx_preset.gd (20 lines), this Resource class defines all visual parameters for a ship's VFX system. Multiple presets can be created and assigned to different ships or factions.

    Properties¶

    Property Type Default Description
    thruster_scene PackedScene null Custom thruster particle scene (optional override)
    booster_scene PackedScene null Custom booster particle scene (optional override)
    throttle_curve Curve new Curve() Custom throttle-to-emission mapping curve
    emission_base float 20.0 Base emission intensity multiplier
    emission_boost_multiplier float 5.0 Emission multiplier during boost
    glow_color Color (1.0, 0.9, 0.7) Normal thrust glow color (warm white)
    boost_glow_color Color (1.0, 0.4, 0.1) Boost-mode glow color (orange)
    glow_energy float 50.0 Base glow energy for light/mesh emission
    boost_glow_energy_multiplier float 2.5 Additional energy multiplier during boost
    particle_scale float 1.0 Base particle size scale
    boost_particle_scale_multiplier float 1.4 Particle scale multiplier during boost
    min_emission float 0.15 Minimum emission to keep VFX visible at idle
    boost_visual_threshold float 0.7 Throttle threshold for visual boost transition
    lod_near_distance float 50.0 Near LOD distance (unused, kept for compatibility)
    lod_mid_distance float 200.0 Mid LOD distance — full → half emission
    lod_far_distance float 600.0 Far LOD distance — half → minimal emission
    lod_cull_distance float 800.0 Cull distance — beyond this, VFX disabled

    Throttle Curve¶

    The throttle_curve: Curve property allows designers to create non-linear throttle responses. For example, a curve that ramps quickly at low throttle and flattens near full throttle creates more responsive-feeling thrust without changing actual flight physics. If no curve is set or it has zero points, the mapping is identity (output = input).

    FlightVfxSync — Integration Helper¶

    Located at /home/j/3d-space/scripts/flight/flight_vfx_sync.gd (33 lines), this RefCounted class provides a static helper for synchronizing VFX state from flight controllers. It's used by both the player FlightController and NPC AI flight controllers to update ship VFX each frame.

    static func sync_ship_vfx(
    ship_vfx: ShipVFX,
    throttle_value: float,
    boost_active: bool,
    include_debug_log: bool,
    vfx_debug: bool,
    vfx_debug_last_msec: int,
    controller_name: String,
    raw_throttle: float
) -> Dictionary

    Returns a dictionary with debug logging status and VFX metrics (socket count, last emission, LOD multiplier). Debug logs are rate-limited to once per second.

    Integration Points¶

    FlightController → ShipVFX¶

    The player FlightController calls FlightVfxSync.sync_ship_vfx() every physics frame: 

    var result := FlightVfxSync.sync_ship_vfx(
    ship_vfx, throttle_value, boost_active,
    include_debug_log, vfx_debug, _vfx_debug_last_msec,
    "PlayerController", raw_throttle
)

    ShipSystemsCoordinator → VFX Configuration¶

    ShipSystemsCoordinator.configure_from_ship_data() calls parent.call("configure_ship_vfx", sdata) on the ship node, which applies the preset from ShipData.vfx_preset.

    RemoteShipController (Multiplayer)¶

    Remote ships receive VFX state through snapshot data. The RemoteShipController applies throttle and boost values to remote ship instances' ShipVFX nodes, ensuring visual consistency across all clients.
