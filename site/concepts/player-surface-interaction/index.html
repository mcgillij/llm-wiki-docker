    Player Surface Interaction System¶

    Summary¶

    The player surface interaction system enables on-foot exploration and interaction with world objects while the player is landed on a planetary body. SurfaceInteractService scans for nearby interactable POIs (surface landmarks, cities, generic POIs) within a configurable distance, manages HUD prompts to prevent UI conflicts, and opens information popups when the player interacts. The system bridges PlayerAvatarController (the on-foot avatar) with world objects through group-based scanning and metadata-driven display names. It handles popup lifecycle management, radial menu integration, and integrates with individual instrument scripts for science activities like deploying gravity balls or thermal penetrators.

    Key Components¶

    SurfaceInteractService — Proximity Scanner and Popup Manager¶

    class_name SurfaceInteractService, extends Node. This is the central coordinator for surface interactions. It runs a periodic scan loop that finds the nearest interactable object within range and manages the interaction popup lifecycle.

    Configuration: - interact_distance: 25.0m — maximum distance to detect interactables - scan_interval: 0.3s — time between proximity scans (throttled accumulator) - target_groups: [&"nav/surface_landmark", &"nav/surface_city", &"nav/poi"] — groups scanned for candidates

    State: - _local_avatar: Reference to the current PlayerAvatarController - _nearest_target: The closest valid interactable node (or null) - _nearest_distance: Distance to nearest target - _popup: The interaction popup CanvasLayer instance (lazily created) - _popup_open: Whether a popup is currently displayed

    Scan loop (_process(delta)): 1. Clears target if avatar is invalid 2. Skips scanning while radial science menu is active (is_radial_menu_active()) 3. Skips scanning while any other popup is open (checked via Game.is_any_popup_open()) 4. Accumulates delta time; when >= scan_interval, calls _scan_nearby() 5. If popup is open, returns early (keeps target set without rescanning) 6. On "fps_interact" key press with a valid nearest target, opens the interaction popup

    Proximity scanning (_scan_nearby()): - Iterates all nodes in target_groups - Computes distance from avatar to each node's global position - Tracks the closest node within interact_distance - When the nearest target changes: sets _local_avatar.set_interact_target(nearest) which updates the HUD prompt; logs the change

    Popup management: - _open_interaction_popup(): Lazily creates the popup via _ensure_popup() (adds to root), calls show_entity(_nearest_target), connects to dismissed signal, sets _popup_open = true - _on_popup_dismissed(): Sets _popup_open = false, clears target so ship re-enter can fire again - _on_radial_menu_opened(): Closes any open popup when the radial science menu opens (prevents conflicts)

    PlayerAvatarController — On-Foot Avatar State Machine¶

    class_name PlayerAvatarController, extends CharacterBody3D. The on-foot avatar has two states: GROUNDED and AIRBORNE. It handles movement, gravity, slope following via Godot's built-in move_and_slide() with floor snap.

    Movement parameters: - Walk speed: 6 m/s, Sprint: 10 m/s, Crouch: 3 m/s - Acceleration: 14 m/s^2, Deceleration: 18 m/s^2 - Air control: 0.4 (reduced horizontal control while airborne) - Jump velocity: 5 m/s, max jumps: 2 - Walkable slope: up to 55 degrees - Coyote time: 0.15s, jump buffer: 0.15s

    Interact target: _interact_target is set by SurfaceInteractService.set_interact_target(). When non-null, the "fps_interact" action is consumed by the service instead of triggering ship re-enter. This prevents accidental re-entry while examining a landmark.

    Radial menu state: _radial_menu_active is set when the radial science menu opens and cleared on dismiss or after an action is performed. While active, surface interaction scanning is paused to prevent UI conflicts.

    Science Action Integration¶

    The avatar's perform_science_action(index) method maps radial menu selections to specific instruments:

    Index Action Condition
    0 Spawn gravity ball GROUNDED + valid spawn location
    1 Spawn thermal penetrator GROUNDED + valid spawn location
    2 Activate spectrometer Always (opens overlay)
    3 Measure radiation Always (opens overlay)
    4 Spawn gas sniffer GROUNDED + valid spawn location
    5 Open planet science dossier Always (opens overlay)
    6 Spawn seismic thumper GROUNDED + valid spawn location
    7 Start lidar scan Always (opens overlay)
    8 Spawn magnetometer drifter GROUNDED + valid spawn location

    Each action either spawns a physical instrument object in the world (gravity ball, thermal penetrator, etc.) or opens an HUD overlay widget. The spawned instruments have their own scripts (GasSniffer, ThermalPenetrator, etc.) that handle scanning and emit results back to the avatar via signals.

    Gravity and Movement on Planetary Surfaces¶

    The avatar attaches to a planetary surface via attach_to_surface_frame(), which sets up: - _attached_frame: A CelestialFrame providing body-local coordinate transforms - _surface_up: The local upward direction (planet radial) - _local_position: Position in the planet's body-local space

    Movement uses Godot's built-in physics with floor snap (apply_floor_snap()) to follow terrain. The avatar's collision shape is a capsule configured via standing_height (1.8m) and capsule_radius (0.35m). Two separate collision layers are used: - COLLISION_LAYER_SHIP_TERRAIN (layer 17): For ship-only terrain patches; avatars do NOT collide with this - COLLISION_LAYER_AVATAR_TERRAIN (layer 18): For avatar-only fine walking collision

    HUD Prompt Management¶

    The FpsInteractPromptWidget displays "Press F — {display_name}" prompts. It receives the interact target from SurfaceInteractService.set_interact_target() and updates its text accordingly. The prompt is hidden when: - No target is in range (target cleared) - A popup is open (interaction consumed by popup) - The radial science menu is active

    Popup Lifecycle¶

    The interaction popup (fps_interaction_popup.gd) follows this lifecycle:

    1. Open: SurfaceInteractService._open_interaction_popup() creates the popup if needed, calls show_entity(_nearest_target) to populate it with entity data from metadata
    2. Active: Target remains set on avatar (prevents ship re-enter); radial menu is blocked; other popups are ignored
    3. Dismiss: Popup emits dismissed signal; service clears target and sets _popup_open = false; next "fps_interact" press can trigger ship re-enter

    Relationships¶

    • Consumes player-possession-system for avatar lifecycle (PlayerPossession.local_avatar_changed)
    • Integrates with fps-hud-instruments for prompt display and instrument overlays via FpsHudOverlay
    • Uses surface-features-generation landmarks as interactable targets (group nav/surface_landmark)
    • Coordinates with moon-landing-surface-interaction for surface frame attachment and gravity handling
