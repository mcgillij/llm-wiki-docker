    Backdrop System — Planets, Nebulae, Landable Surfaces & Sky¶

    Overview¶

    The backdrop system provides all non-interactive world scenery in 3dSpace: distant planets rendered as billboarded spheres, atmospheric effects, nebula clouds, landable planet surfaces with chunked LOD terrain, and reactive sky rendering. It operates independently from gameplay systems — its sole purpose is visual representation of the game world's celestial bodies and environmental atmosphere.

    The system follows a preset-driven architecture where BackdropPreset resources define all visual parameters (billboards, environment, sun direction/color), and individual planet nodes apply their own shaders based on type. Landable planets use a separate chunked terrain system inspired by the cuberact quadtree sample.

    Backdrop — Main Controller¶

    Located at /home/j/3d-space/scripts/backdrop/backdrop.gd (779 lines), Backdrop is a Node3D that serves as the central coordinator for all backdrop rendering. It manages three subsystems: billboards (distant objects), environment (WorldEnvironment with sky shader), and sun (directional light + billboard sun).

    Architecture¶

    Backdrop (Node3D)
    ├─ Billboards root → MeshInstance3D per preset entry (QuadMesh + unshaded material)
    ├─ WorldEnvironment → runtime-duplicated Environment resource with sky shader params
    ├─ DirectionalLight3D → follows sun_direction from preset
    ├─ BillboardSun (BillboardSunGenerator) → billboarded sun disc with glow
    └─ BackdropHeroController → procedural hero manifest for phenomena billboards

    Key Properties¶

    Property Type Default Description
    preset BackdropPreset null Visual preset defining all backdrop parameters
    billboard_distance float 30,000 Distance for billboard placement from camera
    lock_billboards_to_camera bool true Billboards follow camera position (always true)
    enable_billboard_sun bool true Billboarded sun disc with glow halo
    billboard_sun_size float 1,600 Sun disc visual radius (increased from 800 for larger stars)
    reactive_sky bool true Connect to SkyParameterService for dynamic sky updates

    Billboard System (_apply_billboards)¶

    Creates billboard MeshInstance3D nodes from preset entries: 

    for data in preset.billboards:
    mesh_instance = QuadMesh(billboard_data.size)
    material = StandardMaterial3D(
        shading_mode=UNSHADED,
        transparency=ALPHA,
        depth_draw=DISABLED,
        blend_mode=MIX,
        albedo_texture=_get_billboard_texture(data.texture),
        emission=data.color,
        billboard_mode=ENABLED if data.billboard else DISABLED
    )

    Texture processing (_get_billboard_texture): Converts source textures to alpha-masked versions using luminance-based thresholding: alpha = pow(luminance, 1.7). Results are cached in _billboard_texture_cache per-source.

    Camera tracking: Billboards root node follows camera position with a 1-unit move threshold to avoid unnecessary updates. Each billboard stores its direction as metadata and is positioned at direction × effective_distance from the camera.

    Environment & Sky Shader (_apply_environment, _apply_sky_shader_params)¶

    Applies preset environment to WorldEnvironment: 

    runtime_env = GRAPHICS_RUNTIME_RESOURCES.duplicate_environment_for_runtime(preset.environment)
world_env.environment = runtime_env

    Sky shader parameters are set directly on the sky's ShaderMaterial: | Parameter | Source | Description | |-----------|--------|-------------| | sun_dir | preset.sun_direction | Sun direction vector | | sun_color | preset.sun_color | Sun color | | sun_intensity | preset.sun_energy | Sun brightness | | sun_disc_size | preset.sun_disc_size | Disc visual size | | sun_halo_size | preset.sun_halo_size | Halo radius | | sun_halo_intensity | preset.sun_halo_intensity | Halo brightness |

    Reactive Sky Integration (_connect_reactive_sky)¶

    When reactive_sky is enabled, the backdrop connects to SkyParameterService.parameters_changed signal. On each change: 1. _apply_reactive_sky_params(params) applies new sky parameters via SKY_SHADER_MATERIAL_APPLIER.apply_to_shader_material() 2. Debug logging records all shader parameters for troubleshooting

    Billboard Sun (_initialize_billboard_sun, set_billboard_sun_data)¶

    Creates a BillboardSunGenerator instance that renders the sun as a billboarded disc: 

    _billboard_sun.configure(star_data, sun_position)
_billboard_sun.set_billboard_active(true)

    The sun is configured with spectral class data (color, size) and positioned at the star's world position. It uses billboard_sun_size (1,600 default) for disc radius and billboard_sun_glow_radius (400 default) for glow halo.

    Hero Controller (_ensure_hero_controller, apply_procedural_hero_manifest)¶

    Manages procedural hero billboards (phenomena, stations, etc.) via a separate controller node: 

    _hero_controller.apply_manifest(manifest)  # Apply procedural hero data
_hero_controller.clear_manifest()           # Clear all procedural billboards

    Debug System¶

    Extensive debug logging with configurable tags: - BACKDROP_DEBUG_TAG — General backdrop operations - DEBUG_MAP_BRIGHTNESS_TAG — Full environment parameter dump (exposure, tonemap, glow, fog, light energy, shadow bias) - SHADER_DEBUG_TAG — Sky shader parameter parity checks across scenes

    PlanetNode — Planet Rendering¶

    Located at /home/j/3d-space/scripts/backdrop/planet_node.gd (2,431 lines), PlanetNode is a Node3D that renders individual planets with type-specific shaders, clouds, atmosphere, rings, and moons. It's the primary visual representation of celestial bodies in the game world.

    Shader Types¶

    const SHADER_TYPE_ROCKY: int = 0      # procedural_planet.gdshader
const SHADER_TYPE_GAS_GIANT: int = 1  # gas_giant.gdshader
const SHADER_TYPE_ICE_WORLD: int = 2  # ice_world.gdshader
const SHADER_TYPE_LAVA_WORLD: int = 3 # lava_world.gdshader
const SHADER_TYPE_OCEAN: int = 4      # procedural_planet.gdshader (ocean variant)
const SHADER_TYPE_DESERT: int = 5     # desert_world.gdshader
const SHADER_TYPE_TOXIC: int = 6      # toxic_world.gdshader
const SHADER_TYPE_BARE: int = 7       # procedural_planet.gdshader (bare rock)

    Visual Components¶

    Each planet node contains up to 5 visual layers: 1. Planet body — SphereMesh with type-specific shader material 2. Cloud layer — Slightly larger sphere with procedural_clouds.gdshader (rotates independently) 3. Atmosphere — Outer shell with atmosphere.gdshader (Fresnel-based rim lighting) 4. Rings — Torus-like mesh with planet_ring.gdshader (only for gas giants with rings) 5. Distant indicator — Billboarded sprite for planets outside render distance

    LOD System (lod_upgrade_distances, lod_downgrade_distances)¶

    Three-tier LOD based on camera distance: | Tier | Upgrade Distance | Downgrade Distance | Effect | |------|-----------------|-------------------|--------| | 0 (full) | — | >12,000 units | Full detail with clouds, atmosphere, rings | | 1 (medium) | <5,000 | >6,000 | Reduced geometry, no atmosphere | | 2 (low) | <2,000 | — | Minimal sphere, no extras |

    LOD is automatically managed based on camera distance. auto_upgrade enables seamless transitions between tiers.

    Moons & Satellites¶

    Each planet can have moons and satellites orbiting it: - Moons: Small spheres orbiting the planet with configurable speed (MOON_ORBIT_SPEED_SCALE = 0.00025) - Satellites: Pre-built scenes (habitat, mining platform, defense, relay, derelict) orbiting at SATELLITE_ORBIT_SPEED_SCALE = 0.018

    Orbit debug visuals can be enabled to show orbital paths and markers: - Moon markers: Blue spheres with radius 180 units - Satellite markers: Orange spheres with radius 70 units

    Landable Planet Integration¶

    When a planet is marked as landable, the PlanetNode integrates with the landable surface system: 

    LANDABLE_PLANET_ADAPTER_SCRIPT — bridges PlanetNode → LandablePlanetRuntime
PLANET_RUNTIME_BUILD_QUEUE_SCRIPT — manages async terrain build queue
PLANET_SURFACE_MANAGER_SCRIPT — surface generation management

    The adapter handles coordinate space conversion between the planet's orbital position and the landable surface's local origin.

    Collision & Physics¶

    • Collision body: StaticBody3D with SphereShape3D for planetary collision detection
    • Collision radius offset: Configurable via collision_radius_offset export property
    • Surface debug label: Optional Label3D showing planet name and type above the surface

    LandablePlanetRuntime — Chunked Terrain System¶

    Located at /home/j/3d-space/scripts/backdrop/landable/landable_planet_runtime.gd (858 lines), this Node3D implements a quadtree-based chunked terrain system for rendering landable planet surfaces. It's adapted from the cuberact LOD sample and uses a cube-face projection approach (6 root quads, one per face of a cube inscribed in the sphere).

    Configuration¶

    Property Default Description
    radius 6,378 km Planet radius (Earth-like)
    terrain_height 16 km Peak terrain displacement above ideal sphere
    atmosphere_height 222 km Atmosphere extent above surface
    grid_size 16 Vertices per chunk edge (actual grid: 17×17)
    split_budget 8 Max chunk splits per frame
    lod_threshold_deg 10° Screen angle threshold for splitting
    max_lod_level 16 Deepest allowed quadtree level

    Quadtree Structure¶

    Root (6 faces) → Quad (per face) → Chunk (grid_size×grid_size vertices)
                                              ↓ split_budget per frame
                                         Child Quads (4x children)
                                              ↓
                                         Leaf Chunks (rendered meshes)

    Each quad represents one face of a cube inscribed in the planet sphere. When a chunk's screen angle exceeds lod_threshold_deg, it splits into 4 child quads (each with its own grid). This continues up to max_lod_level deep, creating fine detail near the camera and coarse geometry at distance.

    Incremental Build¶

    Terrain generation is incremental — only split_budget chunks are split per frame, preventing frame drops during planet approach: 

    func _process(delta):
    for i in range(split_budget):
        if pending_splits.is_empty(): break
        var quad = pending_splits.pop_front()
        quad.split()  # Creates 4 child quads with new grids

    Child chunk builds are also budgeted: split_child_build_budget (default 1) chunks are built per frame for each split. This ensures smooth loading even at high LOD levels.

    Terrain Generation¶

    Each chunk's vertices are displaced based on procedural noise: 1. Base sphere: Ideal spherical surface at radius km 2. Terrain displacement: Noise-based elevation from 0 (ocean floor) to terrain_height (highest peak) 3. Ocean layer: Separate sphere at reduced radius for water coverage 4. Atmosphere: Outer shell with atmospheric scattering shader

    Shader Materials¶

    • Terrain: landable_terrain.tres — per-pixel elevation coloring based on height bands
    • Ocean: landable_ocean.gdshader — animated wave displacement with Fresnel reflections
    • Atmosphere: Atmospheric scattering with sun-direction-dependent coloring

    Culling & Performance¶

    • Frustum culling: Enabled by default, skips chunks outside camera view
    • Horizon culling: Skips chunks below the visible horizon
    • Custom AABB: All chunks share a single AABB covering the entire planet sphere + terrain height, disabling Godot's built-in frustum culling (which would be too coarse for chunked geometry)

    Collision Guard¶

    collision_guard_enabled prevents the player from falling through the surface: 

    if player_height < collision_guard_height_ratio * planet_radius:
    # Push player up to maintain minimum altitude above terrain

    PlanetSurfaceManager — Surface Generation Management¶

    Located at /home/j/3d-space/scripts/backdrop/planet_surface_manager.gd (519 lines), this @tool Node3D manages the generation of planet surface patches with async support. It's used for non-landable planets that still need surface detail for visual representation.

    Patch System¶

    Surface is divided into latitude/longitude patches: 

    _patches = {  # Dictionary keyed by (lat_index, lon_index)
    (0, 0): {"vertices": [...], "indices": [...], "material": StandardMaterial3D},
    ...
}

    Each patch is a spherical segment with its own geometry and material. Patches are generated asynchronously when async_generation is enabled.

    Signals¶

    Signal Parameters Description
    generation_started — Async generation began
    generation_progress progress (0–1) Generation percentage
    generation_completed — All patches generated
    generation_failed error (String) Generation error message

    LOD Support¶

    When enable_lod is true, the manager adjusts patch resolution based on camera distance to a target node (lod_target_path). Updates occur at lod_update_interval seconds.

    BackdropNebulaData — Nebula Configuration¶

    Located at /home/j/3d-space/scripts/backdrop/backdrop_nebula_data.gd, this Resource defines nebula visual data: | Property | Type | Description | |----------|------|-------------| | scene | PackedScene | Nebula scene to instantiate | | position | Vector3 | World position in the system | | rotation_degrees | Vector3 | Rotation Euler angles | | scale | Vector3 | Scale multiplier |

    Nebulae are instantiated by the Backdrop's _apply_nebulae() method into a nebula root container. Each nebula scene typically contains volumetric fog, particle systems, and emissive meshes for gas cloud visualization.

    BackdropPhenomenonMapper — Phenomena Data Mapping¶

    Located at /home/j/3d-space/scripts/backdrop/backdrop_phenomenon_mapper.gd, this script maps procedural phenomenon data to visual billboard representations. It works with the BackdropHeroController to create dynamic billboards for: - Station clusters - Fleet formations - Anomaly markers - Navigation beacons

    PlanetSkybox — Skybox Rendering¶

    Located at /home/j/3d-space/scripts/backdrop/planet_skybox.gd, this component handles skybox rendering for planetary surfaces. It manages: - Star field background (static star positions) - Sun/moon positioning in the sky dome - Atmospheric scattering effects near the horizon - Day/night cycle transitions

    The skybox uses a custom shader that applies atmospheric color gradients based on sun direction and observer altitude.

    Integration Points¶

    Backdrop → SystemGenerator¶

    When a system is generated, SystemGenerator provides star data (StarData) to the backdrop via set_billboard_sun_data(star_data, sun_position), which configures the billboard sun with the correct spectral class color and size.

    PlanetNode → ProceduralLevelBuilder¶

    The ProceduralLevelBuilder.build_planets() method creates PlanetNode instances for each planet in a system recipe, applying the appropriate shader type based on planet classification (rocky, gas giant, ice world, etc.).

    LandablePlanetRuntime → PlanetNode¶

    When a planet is marked as landable, the LandablePlanetAdapter bridges between PlanetNode's orbital position and LandablePlanetRuntime's local surface coordinate space. The adapter handles: - Position offset (planet center → surface origin) - Rotation alignment (orbital frame → surface frame) - Scale conversion (game units → km for terrain generation)

    Backdrop → SkyParameterService¶

    The reactive sky system connects to SkyParameterService for dynamic updates during gameplay. When the player travels between systems or time of day changes, new sky parameters are applied to the shader material in real-time.
