    Feed System — News Event Broadcast Pipeline¶

    Summary¶

    The feed system is the game's centralized event broadcast and persistence layer. It collects gameplay events from multiple sources (discovery publications, faction reputation changes, route activity, world state shifts), enriches them with metadata, assigns priority scores for story ranking, persists everything to SQLite, and generates daily rollup summaries. Events flow through a deterministic pipeline: raw event dictionary → FES record method → database insertion into feed_event_log and feed_story_candidates tables → optional daily rollup rebuild. UI widgets consume the stored events to display news feeds, headlines, and summaries to players.

    Architecture Overview¶

    ┌──────────────┐    ┌──────────────┐    ┌───────────────┐    ┌──────────────┐
│  GAMEPLAY    │    │   FES        │    │   SQLite DB   │    │     UI       │
│  SOURCES     │───▶│ (event bus)  │───▶│               │───▶│  WIDGETS     │
│              │    │              │    │ feed_event_log│    │              │
│ Discovery    │    │ record_      │    │ feed_candidates│   │ headlines    │
│ Publication  │    │ discovery_   │    │ feed_rollups  │    │ summaries    │
│ Faction rep  │    │ faction_     │    │               │    │ route events │
│ Route events │    │ route_       │    │               │    │ world state  │
│ World state  │    │ world_state_ │    │               │    │              │
└──────────────┘    └──────────────┘    └───────────────┘    └──────────────┘

    Key Components¶

    FES (FeedEventService) — Central Event Bus¶

    class_name FES, loaded as an autoload singleton at /root/FeedEventService. It provides four public record methods, each accepting a Dictionary event payload and returning a result dict with ok, event_id, candidate_id, and status flags. All methods require ServerDatabase.fetch() to be non-null (server-side only).

    Event scope: All events use EVENT_SCOPE_PUBLIC = "public".

    Event Types¶

    Constant Value Source
    EVENT_TYPE_DISCOVERY_PUBLICATION "discovery_facet_published" First-time discovery of a subject/facet
    EVENT_TYPE_DISCOVERY_VALIDATION "discovery_facet_validated" Subsequent validation of existing discovery
    EVENT_TYPE_FACTION_REPUTATION "faction_reputation_swing" Faction standing changes

    Route event types (validated against ROUTE_EVENT_TYPES array): - "delivery", "mining_delivery", "pirate_pressure", "pirate_eliminated", "patrol_response", "trader_destroyed"

    World state event types: - "control_shift", "border_pressure_spike", "frontier_census"

    Discovery Publication Pipeline¶

    record_discovery_publication(event) handles the most complex event type. The pipeline:

    1. Validation: Checks for required subject_id and facet_key; returns early with error if missing
    2. Timestamp resolution: Uses event_ts or falls back to system time; resolves date text
    3. Coordinate resolution: Parses system_coords (Vector3i or Dictionary) into X/Y/Z integers; generates display coordinates like "(1,2,3)"
    4. Facet lookup: Queries DiscoveryFacetRegistry.get_definition(subject_kind, facet_key) to determine if the discovery is newsworthy and get its category
    5. Story classification: Resolves story_kind (e.g., "science_breakthrough", "anomaly_alert", "frontier_milestone") based on facet key, first-publication flag, and category
    6. Priority scoring: _compute_discovery_story_weight() starts at base 40, adds: +25 if newsworthy, +20 for first publication, +min(10, validation_count) for extra confirmations, +5 each for system name and subject name presence, +5 for science category
    7. Event ID generation: "evt:{type}:{subject_id}:{facet_key}:{timestamp}" with player suffix for validations
    8. Candidate ID generation: "cand:{date}:{story_kind}:{subject_id}:{facet_key}"
    9. Headline/summary hints: Template-based text generation using _build_discovery_headline_hint() and _build_discovery_summary_hint()

    Faction Reputation Tracking¶

    record_faction_event(event) handles standing changes:

    • Skips events where abs(delta) < 0.2 and no standing threshold was crossed
    • Story kind is "standing_shift" if the standing tier changed, otherwise "reputation_swing"
    • Priority score starts at 45 + min(20, abs(delta)*20), plus +20 for standing tier changes
    • Event ID: "evt:faction:{player_id}:{faction_id}:{timestamp}"

    Route Events¶

    record_route_event(event) processes delivery, mining, and pirate activity:

    • Validates event type against ROUTE_EVENT_TYPES array
    • Categories: "economy" (delivery, mining_delivery) or "security" (pirate events, patrol, trader_destroyed)
    • Story kinds map to narrative types: "trade_report", "mining_report", "security_alert", "patrol_success", etc.
    • Priority scoring incorporates risk level and cargo amount as modifiers

    System Directory Upsert¶

    Both discovery publications and route events call upsert_system_directory_entry() which maintains a live directory of systems with their current faction control, economy type, security profile, and population data. Uses INSERT ... ON CONFLICT to update in place.

    Story Candidate Generation¶

    Every event creates a corresponding entry in the feed_story_candidates table. Candidates are the primary source for UI feed widgets — they contain the pre-computed headline hint, summary hint, priority score, and metadata needed to render news items without re-processing raw events.

    The build_story_candidates_for_date() method can regenerate candidates from existing event log entries (for rebuild scenarios).

    Daily Rollup System¶

    _rebuild_discovery_rollup_for_date() and _rebuild_route_rollup_for_date() aggregate daily statistics into the feed_daily_rollups table:

    Discovery rollup: - Total public discoveries count - First discovery count - Top 3 systems by event count - Top 5 players by candidate count

    Route rollup: - Total deliveries, mining deliveries - Total disruptions (pirate pressure + trader destroyed) - Total pirates eliminated - Total mining quantity - Top 3 systems by route activity

    The rebuild_daily_rollup(date_text) public method triggers both rollups for a given date. Rollups are automatically rebuilt after each discovery or route event insertion via _rebuild_discovery_rollup_for_date() and _rebuild_route_rollup_for_date().

    SQLite Persistence Schema¶

    feed_event_log table columns: - event_id (PK), event_type, event_scope, event_ts, date_text, world_instance_id - player_id, player_tag, system_x/y/z, system_name, display_coordinates - subject_id/kind/entity_id, subject_display_name, parent_subject_id/display_name - faction_id/name, category, story_weight, newsworthy (0/1) - context_json, payload_json, tags_json, metadata_json

    feed_story_candidates table columns: - candidate_id (PK), source_event_id, date_text, story_kind, category - priority_score, headline_hint, summary_hint - system_x/y/z, system_name, display_coordinates - subject_id/kind/display_name, source_player_id/tag, is_first_claim, validation_count - metadata_json

    feed_daily_rollups table columns: - date_text (PK), discovery counts, route counts, mining quantities - top_systems_json, top_players_json - updated_at

    Event Flow Summary¶

    1. Gameplay action occurs (e.g., player discovers a new system)
    2. Source calls FES method with an event dictionary containing all relevant fields
    3. FES validates and enriches: resolves timestamps, coordinates, facet definitions; computes story weight, headline/summary hints
    4. Database insertion: Event goes into feed_event_log; candidate goes into feed_story_candidates (both use upsert for idempotency)
    5. System directory updated (for discovery and route events)
    6. Daily rollup rebuilt automatically after the event
    7. UI widgets query feed_story_candidates ordered by priority score to display ranked news items

    Relationships¶

    • Consumed by discovery-exploration for publication/validation events
    • Integrated with factions-lore via faction reputation tracking in FactionManager
    • Used by NPC simulation (NpcSimulationService) and patrol management (PatrolM) for route/world-state events
    • Depends on fw-config-service for server database configuration
