    Networking & Multiplayer Replication Layer¶

    Overview¶

    The multiplayer system uses an authoritative server model where the server simulates all player ships deterministically and broadcasts thin state snapshots to clients. Clients render remote ships as visual proxies with smooth interpolation between snapshots. The system handles world partitioning (peers are assigned to specific world instances), ship interest management, beam weapon replication, and projectile spawning via MultiplayerSpawner.

    Architecture Overview¶

    ┌─────────────────────┐         ┌──────────────────────┐
│   SERVER            │         │   CLIENT             │
│                     │         │                      │
│ NetworkShipManager  │────────▶│ RemoteShipController │
│ (authoritative sim) │ snapshot│ (visual proxy only)  │
│                     │         │                      │
│ MultiplayerSession  │ input   │ FlightController     │
│ (flight input recv) │◀────────│ (local ship control) │
└─────────────────────┘         └──────────────────────┘

    Server-Side: NetworkShipManager¶

    Located at /home/j/3d-space/scripts/network/network_ship_manager.gd (2,564 lines), this is the central multiplayer coordinator. It runs on both server and clients but has different responsibilities:

    Server Responsibilities¶

    1. Authoritative ship simulation — Reads flight input from each peer, runs SharedFlightModel.simulate(), applies movement
    2. Snapshot broadcasting — Collects ship states into packed binary snapshots, broadcasts to all peers in the same world instance
    3. Beam weapon authority — Determines beam fire state and target positions authoritatively for consistent visual replication
    4. Projectile spawning — Uses MultiplayerSpawner to spawn projectiles with network ownership

    Client Responsibilities¶

    1. Local ship control — Runs own FlightController with local physics (input → movement)
    2. Remote proxy rendering — Creates RemoteShipController instances for each peer, receives and interpolates snapshots
    3. Input submission — Sends flight input state to server via MultiplayerSession.set_flight_input_state()

    Key Constants¶

    Constant Value Purpose
    SNAPSHOT_INTERVAL 1/30s (33.3ms) Server snapshot broadcast rate
    SNAPSHOT_FORCE_INTERVAL 0.15s Force broadcast even if no change
    SHIP_INTEREST_NEAR_DISTANCE 6,000m Near distance for interest management
    SHIP_INTEREST_FAR_DISTANCE 8,000m Far distance — always replicate
    SHIP_SNAPSHOT_NEAR_DISTANCE 2,500m High-frequency snapshot range
    SHIP_SNAPSHOT_MEDIUM_DISTANCE 6,000m Medium-frequency snapshot range

    Snapshot Delivery Policy¶

    Ships are replicated at different frequencies based on distance from the viewer:

    Distance Range Tick Divisor Effective Rate
    < 2,500m (near) 1 (every frame) ~30 Hz
    2,500–6,000m (medium) 2 ~15 Hz
    > 6,000m (far) 5 ~6 Hz

    Packed Snapshot Format¶

    Snapshots are packed into a binary layout for efficient network transmission:

    Meta stride (5 ints per ship): peer_id, time_ms, flags_bitfield, weapon_type, sequence

    Data stride (23 floats per ship): pos.xyz, quat.xyzw, vel.xyz, throttle, travel_spinup, beam_origin_pos.xyz, beam_target_pos.xyz, shield_current, hull_hp, highway_active, highway_progress

    World Instance Partitioning¶

    Peers are assigned to world instances via MultiplayerSession.get_assigned_world(). The server only replicates ships between peers in the same world instance: 

    var loaded_peer_ids = MultiplayerSession.get_loaded_peer_ids_for_world(world_instance_id)

    Beam Weapon Replication¶

    Beam weapons (lasers) require special handling because they need consistent visual representation across all clients. The server tracks beam state per peer: - fire — Whether the weapon is firing - beam_target_active — Whether a valid target exists - beam_origin_pos — Muzzle position in world space - beam_target_pos — Target position (or aim point if no direct lock)

    A fallback mechanism ensures beams remain visible even if snapshot data is delayed.

    Client-Side: RemoteShipController¶

    Located at /home/j/3d-space/scripts/network/remote_ship_controller.gd (626 lines), this extends ShipVisualProxy and serves as a lightweight visual-only representation of remote player ships. It does NOT run physics or process input.

    Setup¶

    func setup(peer_id: int, ship_data: ShipData, initial_spawn_context: String = "") -> void
    Creates the proxy node, assigns it to groups (ship/remote, ship, nav/player_remote), and builds visual nodes including a jump flash effect for spawn/depart VFX.

    Snapshot Interpolation Pipeline¶

    1. Receive — Server snapshots arrive via receive_snapshot(state: Dictionary) → pushed into SnapshotBuffer
    2. Sample — Each frame, _process() calls _snapshot_buffer.sample(server_time_ms, rtt_ms) to get interpolated state
    3. Apply — Position/rotation from interpolated state are applied directly to the proxy node

    Dynamic Interpolation Delay¶

    The buffer adapts its interpolation delay based on network conditions: - Uses RTT (round-trip time) from MultiplayerSession.get_latency_ms() - Calculates one-way floor as RTT × 0.5 - Target delay = max(expected_interval + jitter_budget, one_way_floor) - Clamped to [40ms, 160ms] range

    Extrapolation¶

    When snapshots are delayed or lost, the buffer extrapolates using the last known velocity: 

    result.position = newest.position + newest.velocity * dt
    Extrapolation is capped at a dynamic budget (2× expected interval + jitter) to prevent visual drift. Three reasons are tracked: "boundary", "late_snapshot", and "extrapolation_cap".

    Weapon Visual State¶

    Remote ships display weapon visuals based on snapshot data: - _apply_weapon_visual_state(fire, beam_target_active, beam_origin_pos, beam_target_pos, weapon_type) - Beam origin/target positions are interpolated between snapshots for smooth beam rendering - Change detection prevents redundant VFX calls

    Spawn/Depart VFX¶

    • Spawn: Plays context-specific visual effect (e.g., jump-in flash)
    • Depart: Shows jump flash, animates fade-out over 0.45s, then queues the node for freeing

    SnapshotBuffer¶

    Located at /home/j/3d-space/scripts/network/snapshot_buffer.gd (460 lines), this is a timestamped ring buffer that stores authoritative snapshots and provides smooth interpolation between them. It's used by both RemoteShipController and AI flight controllers in network replica mode.

    Data Structures¶

    • Snapshot — Single entry: time_ms, position, rotation (Quaternion), velocity, throttle, boost, travel_mode, fire state, beam data
    • InterpolatedState — Result of sample(): interpolated values + is_extrapolating flag + reason string

    Ring Buffer Operations¶

    • push() — Adds snapshot at write index, wraps around buffer size (default 32). Rejects out-of-order snapshots.
    • sample(render_time) — Finds bracketing pair in ring order, interpolates position/velocity with linear lerp and rotation with quaternion slerp

    Interpolation Details¶

    Field Method Notes
    Position Vector3.lerp() Linear interpolation
    Rotation Quaternion.slerp() No gimbal lock
    Velocity Vector3.lerp() Linear interpolation
    Throttle lerp() Float interpolation
    Boost/Travel/Fire Threshold at t ≥ 0.5 Boolean snap decision
    Beam positions Interpolated if both snapshots firing Otherwise snap to later snapshot

    Adaptive Extrapolation¶

    The buffer tracks recent snapshot intervals (rolling window of 8) and uses them to: 1. Calculate expected interval for jitter detection 2. Set dynamic extrapolation budget 3. Track consecutive missed snapshots for diagnostics

    Smoothing¶

    Optional low-pass filter applied during extrapolation only: 

    result.position = last_smoothed.lerp(result, smoothing_factor)
    Factor range: 0.05–0.5 (default 0.1). Resets when no longer extrapolating.

    SharedFlightModel¶

    Located at /home/j/3d-space/scripts/network/shared_flight_model.gd, this provides the deterministic physics simulation that runs on both server and clients. It defines:

    FlightState¶

    class FlightState extends RefCounted:
    var position: Vector3 = Vector3.ZERO
    var rotation: Quaternion = Quaternion.IDENTITY
    var velocity: Vector3 = Vector3.ZERO
    var throttle: float = 0.0
    var boost_active: bool = false
    var travel_mode_active: bool = false
    var travel_mode_spinup_progress: float = 0.0

    FlightInput¶

    class FlightInput extends RefCounted:
    var forward: float = 0.0
    var strafe: float = 0.0
    var rotate_yaw: float = 0.0
    var rotate_pitch: float = 0.0
    var rotate_roll: float = 0.0
    var boost: bool = false
    var fire: bool = false
    var sequence: int = 0

    FlightParams¶

    Contains ship-specific parameters loaded from ShipData: base speed, rotation rates, bank angle limits, etc.

    Simulation Method¶

    static func simulate(ship: FlightController, input: FlightInput, params: FlightParams, state: FlightState, delta: float) -> void
    Applies input to the ship's physics state deterministically — same inputs always produce same outputs regardless of frame timing (within reason).

    ShipReplicationService¶

    An autoload singleton (/root/ShipReplicationService) that manages the lifecycle of ship replication: - Creates RemoteShipController instances when peers join - Removes them when peers leave or disconnect - Handles world assignment changes (peer moves to different world instance) - Coordinates with NetworkShipManager for spawn synchronization
